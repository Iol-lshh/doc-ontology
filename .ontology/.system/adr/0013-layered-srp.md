# ADR 0013: 레이어 분리와 SRP — Controller → Facade → Service

- Status: Accepted
- Date: 2026-06-19

## Context

GUI/CLI는 동일 엔진을 호출하는 대칭 프론트엔드다([[0011]]). 엔진이 가진 일은 하나가 아니다 — 빌드([[0008]]), 롤백([[0010]]), 조회([[0007]]), 비교([[0012]]). 이걸 한 덩어리(`core`, `build-ontology`)로 두면 두 가지가 무너진다.

- **이름이 책임을 참칭한다.** `build-ontology` 한 파일이 롤백·조회·비교까지 담으면 이름과 내용이 어긋난다.
- **의존성 방향이 거꾸로 선다.** 명령(build-ontology 등)은 더 작은 기능(스캔·검증·인덱스·스냅샷)을 **조율**하는 쪽이다. 조율자에게 가장 안쪽 이름(`core`)을 붙이면, 가장 많이 의존하는 것이 가장 안쪽이 되어 방향이 뒤집힌다.

## Decision

세 레이어로 나누고, **의존은 항상 바깥에서 안쪽(도메인)으로만 흐른다.** 안쪽은 바깥을 모른다.

```
바깥 ──의존──▶ 안쪽
CliCommandController / GuiCommandController   진입 + 어댑터
        │  (명령 로직 없음, 알맞은 Facade 호출)
        ▼
*Facade                                       유스케이스 조율 (순서·정책)
        │
        ▼
~Service                                      단일 책임 (스캔·검증·인덱스·스냅샷 …)
```

**Controller** — 채널(CLI/GUI)별 진입점이자 어댑터. CLI는 argv 파싱·콘솔 출력, GUI는 HTTP·하트비트([[0011]])를 진다. 명령 로직은 갖지 않고 Facade로 위임한다. 두 Controller는 채널만 다르고 같은 Facade를 부른다(진입 비대칭·엔진 대칭, [[0011]]).

**Facade** — 한 유스케이스를 조율한다. 여러 Service를 순서·정책대로 엮어 하나의 명령을 완성한다(예: build = 스캔→검증→인덱스 쓰기를 "통과 시에만 커밋", [[0008]]).

**Service** — 단일 책임. 스캔·판정([[0005]]), 검증([[0008]]), 인덱스 3종([[0007]]), 스냅샷([[0010]]) 등. 도메인 기능이며 가장 안쪽이라 아무것도 의존하지 않는다.

### SRP — 파사드는 읽기 소스/유스케이스당 하나

파사드 하나는 변경 이유 하나만 가진다. 유스케이스가 갈리거나(쓰기/롤백/비교) 읽기 소스가 다르면(조회의 탭) 파사드를 나눈다.

| Facade | 책임 |
|---|---|
| `BuildFacade` | 스캔·판정·검증·빌드([[0008]]) |
| `RollbackFacade` | 롤백 — 빌드 롤백 / 히스토리 롤백([[0010]]) |
| `DiffFacade` | 스냅샷 비교 — 그래프 diff + 파일 diff([[0012]]) |
| `FindFilePathFacade` | fileIndex 전체 → 파일 구조도(트리, [[0005]]) |
| `FindGraphFacade` | graphIndex + labelIndex → 관계 그래프([[0007]]) |
| `FindHistoryFacade` | 스냅샷 이력 → 세대 목록·스냅샷([[0010]]) |
| `FindDocumentFacade` | fileIndex로 단건 해소 → 노드 본문 로드([[0001]]) |

`Find`는 단일 유스케이스가 아니라 서로 다른 읽기의 묶음이었다. 구조도(전체 순회)·관계 그래프(인덱스 조율)·세대(스냅샷)·본문(단건 로드)은 읽는 소스와 변경 이유가 모두 달라 하나의 `FindFacade`로 두면 뚱뚱해진다. 그래서 넷으로 나눈다.

## Consequences

- 각 파일이 변경 이유 하나만 가진다(SRP). 무관한 변경이 한 클래스에 모이지 않는다.
- 의존이 안쪽으로만 흐르므로 Service는 채널·HTTP를 모르고 재사용된다. CLI·GUI가 같은 Facade·Service를 공유한다([[0011]]).
- 명령 로직이 Controller(어댑터)로 새지 않아 두 채널의 동작이 갈리지 않는다.
- 파사드 수가 늘어난다(현재 7개). 단일 책임의 대가다.
